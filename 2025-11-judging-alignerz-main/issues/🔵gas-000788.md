# [000788] Incorrect Loop Starting Index Causes Unnecessary Gas Consumption
  
  ### Summary

An incorrect loop starting index will cause increase gas consumption for the protocol as functions `_setDividends` and `getTotalUnclaimedAmounts` iterate over an invalid NFT ID that always returns false.

### Root Cause

In DividendDistributor.sol, the functions `_setDividends` and `getTotalUnclaimedAmounts` initialize their loops with `i = 0`, however NFT IDs begin at index 1. This causes the loop to always check a non-existent NFT ID on the first iteration, wasting gas on a call to `safeOwnerOf(0)` that will always return false.

[_setDividends](https://github.com/dualguard/2025-11-alignerz/blob/main/protocol/src/contracts/A26ZDividendDistributor/A26ZDividendDistributor.sol#L214-L224)

[_startTokenId](https://github.com/dualguard/2025-11-alignerz/blob/main/protocol/src/contracts/nft/ERC721A.sol#L105-L107)

### Attack Path

This is not an attack, it's a gas optimization.

1. The dividend distribution mechanism calls `getTotalUnclaimedAmounts()` or `_setDividends()`
2. The loop initializes with `i = 0` and checks `safeOwnerOf(0)`
3. Since NFT IDs start at index 1, `safeOwnerOf(0)` returns false for a non-existent token
4. The loop continues to index 1 onwards where valid NFT IDs exist
5. One full iteration of unnecessary gas computation is wasted per function call

### Impact

The protocol suffers an approximate loss of 2,000-3,000 gas per function call due to the wasted iteration checking a non-existent NFT ID. For protocols with frequent dividend distributions or queries, this compounds to significant unnecessary gas expenditure over time.

### Mitigation

Change the loop starting index from 0 to 1 to match the ERC721A token ID range.
